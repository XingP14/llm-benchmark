## Product Direction

LLM Benchmark is a local and self-hostable LLM evaluation tool. It provides:

1. CLI benchmark execution across multiple providers.
2. Five stable v0.4.0 dimensions: dialogue, coding, function calling, long
   context, and multi-turn consistency.
3. Report generation in JSON, Markdown, and HTML.
4. Web UI with authentication, SQLite persistence, and WebSocket progress.
5. Docker deployment for self-hosted evaluation runs.

The current project priority order is:

1. Keep CI green and the package buildable.
2. Keep v0.4.0 runtime behavior stable while v0.5.0 benchmark extensions are
   developed behind clear type/config surfaces.
3. Resolve release drift: source says `0.4.0`, but public npm/release/tag state
   does not currently prove `0.4.0` is published.
4. Keep docs aligned across `README.md`, `README.en.md`, `CHANGELOG.md`,
   `ROADMAP.md`, and package metadata.
5. Convert roadmap-only benchmark ideas into tested runtime behavior before
   adding more roadmap volume.

## Release Cadence

LLM Benchmark uses SemVer while still in the `0.x` stage.

| Surface | Version source | Public trigger | Required preflight |
| --- | --- | --- | --- |
| npm package | `package.json` | maintainer `npm publish` | `npm ci && npm run build && npm run lint && npx jest --runInBand --bail --forceExit && npm pack --dry-run` |
| Docker image | `package.json` and Git tag | `.github/workflows/docker.yml` on `v*` tag or default branch | CI green plus Docker build workflow |
| GitHub release | Git tag | maintainer release creation | changelog, tag, npm/Docker evidence |
| CLI | `dist/index.js` generated by build | included in npm package | `npm run build` and CLI smoke test |
| Web UI | `public/` and `dist/web/server.js` | npm/Docker | `npm run build` and `/api/health` smoke test |

Cadence policy:

- Automated commits may run daily.
- Patch releases should contain verified bug, CI, package, doc, or dependency
  fixes with no API break.
- Minor releases should batch new adapters, scoring dimensions, or Web UI
  capabilities with matching docs and tests.
- Do not cut a release from a red CI state, a dirty working tree, or mismatched
  package/tag/changelog state.
- Do not publish from a local shell unless the maintainer explicitly approved
  that release.

## Requirements Rules

Every non-trivial change needs a clear requirement before implementation:

- problem statement: what benchmark, provider, report, API, or Web UI workflow is
  broken or improved
- affected module: `adapter`, `benchmark`, `scoring`, `evaluator`, `reporter`,
  `web`, `cli`, `data`, `ci`, `docs`, or `release`
- acceptance criteria: exact command output, score shape, API response, report
  content, or package contents expected after the change
- compatibility: whether the change affects config JSON, output schema, DB
  schema, CLI flags, Docker runtime, or Web API
- rollback: how to revert if CI, runtime, or package publication fails

Do not start from vague prompts like "add leaderboard compatibility" unless it is
linked to a concrete benchmark config field, runtime path, test, package, or
release task.

## Design Rules

Small fixes need:

- one focused diff
- one narrow test or build command
- doc/changelog update only if behavior, install, API, scoring, Docker, or
  release metadata changed

Cross-cutting changes need a design note before code when they touch:

- scoring algorithms or benchmark dimensions
- provider adapter contracts
- output report schema
- Web API or WebSocket message schema
- SQLite schema or migrations
- auth, secrets, sandbox execution, or Docker runtime
- release workflow or package contents

## Development Rules

- Prefer module-local changes. Do not edit every doc and type file unless the
  change is genuinely cross-cutting.
- Keep generated output out of source control unless the package release requires
  it. Avoid committing `coverage/`, `data/`, `memory/`, `results/`,
  `test-data/`, `node_modules/`, local configs, and temporary archives.
- Use structured JSON parsing/editing for configs and package metadata.
- Pair benchmark/scoring changes with tests that prove the score shape and edge
  cases.
- For new adapters, add provider-specific tests and document environment/config
  requirements.
- For new Web API or CLI behavior, update both Chinese and English README files
  when user-facing behavior changes.
- Keep commit subjects concise:
  - good: `fix(web): validate evaluation id before websocket start`
  - good: `test(scorer): cover function calling argument mismatch`
  - bad: multi-paragraph subjects with full verification logs

## Testing Matrix

Run the narrowest relevant command first, then broaden when a change crosses
module boundaries.

| Changed area | Required verification |
| --- | --- |
| `src/types/**` | `npx tsc --noEmit -p tsconfig.json` |
| `src/adapters/**` | `npm run build` and `npx jest tests/adapter.test.ts --runInBand` |
| `src/benchmarks/**` | `npm run build` and `npx jest tests/benchmarks.test.ts --runInBand` |
| `src/core/scorer.ts` | `npm run build` and relevant scorer/evaluator tests |
| `src/core/evaluator.ts` | `npm run build` and relevant evaluator tests |
| `src/core/reporter.ts` | `npm run build` and `npx jest tests/reporter.test.ts --runInBand` |
| `src/sandbox/**` | `npm run build` and `npx jest tests/sandbox.test.ts --runInBand` |
| `src/web/**` | `npm run build` and relevant `tests/web/*.test.ts` |
| `public/**` | Web UI smoke test after `npm run build` when behavior changes |
| `.github/workflows/**` | validate workflow trigger, install command, and cache behavior |
| docs only | `git diff --check`; run code tests only if examples imply behavior |
| release/package metadata | `npm run build`, `npm pack --dry-run`, version/tag/README/changelog consistency check |

Before declaring work complete, record exactly which commands ran and whether
they passed. If a command was skipped, record why.

## CI And Failure Policy

CI failures outrank new work.

When CI fails:

1. Fetch the failing workflow run and job log.
2. Classify it as code bug, test gap, workflow issue, package/lock issue,
   registry/network issue, Docker issue, or external service issue.
3. Reproduce locally when possible.
4. Fix root cause, not only symptoms.
5. Update `docs/ci-failures.md` when the failure is non-trivial or supersedes a
   prior note.
6. Push only after local verification and watchdog gate pass.

## OpenClaw Automation Rules

OpenClaw cron agents must obey this project-specific loop:

1. Read `AGENTS.md`, this document, `docs/TESTING_STANDARD.md`, and today's
   local runtime note if one exists outside source control.
2. Check CI before selecting work.
3. Prefer the highest-priority real-code candidate from
   `/usr/local/bin/heartbeat-watchdog.sh hint llm-benchmark`.
4. Do not produce more than two roadmap-only commits per local day.
5. Do not make `feat(type)`, `feat(docs)`, `feat(roadmap)`, `chore(docs)`, or
   generic `docs(*)` commits unless the watchdog gate allows them and they are
   tied to a concrete next code step.
6. Use concise commit subjects and place evidence in the body.
7. If push fails, leave the commit in place, document the failure, and stop
   retrying until transport or remote state is understood.

## Security Rules

- Never commit API keys, model provider credentials, JWT secrets, admin
  passwords, npm tokens, or GitHub tokens.
- Keep local config files ignored. Use `config.example.json` for safe examples.
- Runtime DBs and test DBs are local artifacts and must not enter source control.
- For sandbox changes, document command execution boundaries and test malicious
  input cases.
- Security-sensitive bugs go through GitHub private vulnerability reporting, not
  public issues.

## Documentation Rules

- Keep `README.md` and `README.en.md` structurally aligned for user-facing
  features.
- Keep `CHANGELOG.md` release links consistent with actual tags.
- Do not rewrite historical changelog entries except to mark superseded notes.
- Mark stale coverage or release status explicitly instead of implying it is
  current.
- `ROADMAP.md` is planning history and can be long; summarize durable rules in
  governance docs instead of appending more operational policy there.

## Release Checklist

1. Working tree clean except intentional version/changelog/release changes.
2. `git pull --rebase origin master` completed or remote state explicitly
   understood.
3. Package version, changelog, README badges, package manifest, Docker tags, and
   Git tag agree.
4. Required testing matrix commands passed.
5. `npm pack --dry-run` confirms intended files are included.
6. GitHub Actions on target commit are green.
7. Maintainer approved release.
8. Create and push the SemVer tag.
9. Verify npm package, Docker image, and GitHub release artifacts.
10. Record release evidence in `CHANGELOG.md`.
